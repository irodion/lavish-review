# The skill ships self-contained and installs on all SKILL.md platforms; setup is a first-run script

The skill has always lived at `.claude/skills/branch-review-cockpit/`, but its scripts are thin shims that import the real, tested logic from this repo's `src/branch_review/` — four directory levels up. That layout is right for development (one source of truth, one test suite) and fatal for distribution: the `skills` CLI (`npx skills add`) installs a skill by **copying its directory** into a target repo, and the copy has no `src/` to import. Meanwhile the ecosystem has converged: Claude Code, Cursor, and OpenAI Codex all consume the same agentskills.io `SKILL.md` format, Codex and Cursor share the `.agents/skills/` discovery path, and none of them — nor the `skills` CLI — offers a post-install hook. Whatever setup the tool needs beyond the copy must happen some other way.

**Decision (self-contained skill).** The `branch_review` package is **vendored inside the skill directory** (`lib/branch_review/`), committed to the repo. `skills add` copies the source tree as-is — there is no build step to generate an artifact at install time, so the vendored copy must exist in git. Every script shim becomes dual-mode: prefer the repo's `src/` when present (development in this repo — the tests keep pinning one source of truth), fall back to the adjacent vendored copy (an installed skill). Two guards keep the copies honest: `tools/sync_vendored.py` refreshes the vendored tree deterministically, and a test fails whenever vendored and source files diverge — drift is a broken build, not a silent skew.

**Decision (installer as first-run script, not hook).** Setup beyond the copy — the per-machine config with the pinned Lavish version, gitignoring `.review-agent/` and `.lavish-axi/`, per-platform entry points — is `scripts/install.py`, shipped inside the skill and run once by the user (or by the agent on first use when it detects a fresh machine). It is **idempotent** (safe to re-run; never overwrites an existing machine config) and **platform-aware**: it writes what the detected platforms need and nothing else. It follows the house shape — pure planning policy over a thin I/O shell — so what it would do is table-testable without touching a filesystem.

**Decision (platform coverage).** One skill, three platforms, honest degradation where capabilities differ:

- **Entry points.** Claude Code gets `/review-branch`, `/review-resume`, `/review-close` command files plus the `review-analyst` agent definition (both shipped as templates inside the skill and copied out by the installer). Cursor gets the same three commands under `.cursor/commands/`. Codex gets **no files**: skills are invoked natively there (`$`-mention or implicit activation by description), and its custom prompts are user-scoped — the installer does not write into `$HOME` beyond the machine config the user asked for.
- **The isolated analyst ([ADR-0011](./0011-independent-analysis-context.md)) degrades with disclosure.** The isolation boundary is Claude Code's registered subagent. On a platform with an equivalent isolated-context mechanism, the skill uses it under the same input manifest. On a platform with none, the analysis runs in the invoking context and the cockpit's L0 **must say so** — a fixed trusted notice that independence was not enforced by construction. The premise degrades visibly, never silently; this is the host-seam posture ([ADR-0009](./0009-layered-claim-evidence-cockpit.md)) applied to the agent platform instead of the UI host.
- **SessionStart ambient resume stays Claude-Code-only and deferred.** The `sessionstart_hook` config key exists ([ADR-0008](./0008-stdlib-config-loader.md) schema); the hook itself remains v1.1 roadmap. The installer records the preference, nothing more.

**Decision (version pinning).** The pinned Lavish version has exactly one authoritative constant in code (`branch_review.install.PINNED_LAVISH_VERSION`); the installer writes it into the machine config, `SKILL.md` quotes it, and a test asserts the quote matches the constant — the same drift-tripwire pattern as the vendored tree. Lavish is always invoked as `npx -y lavish-axi@<pinned>` so an upstream release can never change behavior mid-review.

**Why vendor rather than restructure.** Moving the package into the skill and making `src/` the shim was considered and rejected: the test suite, typing, linting, and packaging tooling all key off a conventional `src/` layout, and the skill directory is copied verbatim into other people's repos — it should carry only what a review run needs. Duplication is the cheaper cost because it is *mechanically checked*; a restructure would be a permanent ergonomic tax on development to save a generated directory.

## Consequences

- The repo carries `lib/branch_review/` inside the skill — generated, committed, and equality-pinned to `src/branch_review/`. PRs that touch `src/` must re-run `tools/sync_vendored.py` (the failing drift test says exactly this).
- `install.py` is the one place platform differences live; `SKILL.md` gains a short install section and the step-3 analyst ladder, and stays platform-neutral everywhere else (its scripts are already git + stdlib only).
- The command files and agent definition under `.claude/` in this repo become *installed instances* of templates shipped in the skill's `assets/` — the same equality test covers them.
- A `metadata.version` field in the SKILL.md frontmatter becomes the release identity `skills update` operates on; bumping it is part of cutting a release.
- The clean-machine dry-run (issue #12's HITL criterion) is the acceptance gate: `npx skills add` into a scratch repo, `install.py`, run a review — on Claude Code plus at least one of Cursor/Codex.
